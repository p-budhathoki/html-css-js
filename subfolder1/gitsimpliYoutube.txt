Devops :
--------
- Devops is a culture that allows development and operations team to work together
- Developers continuously develop and test codes 
- there is continuously integration taking place throughout the lifecycle
- operatios team continuously deploy the codes to the production environment
- allows better collaboration, increased trust and faster software releases

version control system(VCS) :
-----------------------------
- a snapshot of the files are stored in the remote repository as version 1
- now, there has been some changes to file2 and is updated to file2.1. This change is stored as version 2 in the repository
- VCS allows you to track the history of a collection of files
- each version captures a snapshot of the files at a certain point in time and the VCS allows you to switch between these versions

certralized version control system :
------------------------------------
- it uses a central server to store all the files
- every operation is performed directly on the repository
- all the versions of the files are stored on the Central VCS server
- in case the central server crashes, entire data of the project will be lost. Hence, Distributed VCS was introduced

Git :
-----
- Git is a VCS for tracking changes in computer files. It is generally used for Source Code Management in software development

- git is used to track changes in the source code
- distributed verson control tool used for source code management
- allows multiple developers to work together

features of git :
-----------------
- tracks history
- free and open source
- supports non-linear development
- scalable
- branching is easier
- creates backup
- supports collaboration
- distributed development

git workflow :
--------------
git server --> clone operation --> working copy
git server --> update operation ==> 
working copy --> edit, add, move files --> Modify working copy
Modify working copy --> status and diff operation --> Review changes  -->  commit and push operation --> Commit changes ==> push operation --> git server
Commit changes --> ammend and push operation --> review changes ==> push operation --> git server

branch in git :
---------------
- branch in git is used to keep your changes until they are ready
- you can do your work on a branch while the main branch(master) remains stable. After you are done with your work, you can change it to the main branch

git init ==> create repositories
make changes ==> add, commit and status
sync repositories ==> push, pull and add origin
parallel development ==> branch, merge and rebase

merge and merge conflicts :
---------------------------
- git merge command combines separate branches and resolves any conflicting edits
- a merge conflict is an event that takes place when git is unable to resolve differences in code between the two commits automatically
- git can automatically merge the changes only if the commits are on different lines or branches

merge conflicts when the merge process is started :
---------------------------------------------------
- if there are changes in the working directory of the staging area of current project, merge will fail to start
- in this case conflicts happen due to pending changes which needs to be stabilized using different git command

merge conflicts during the merge process :
------------------------------------------
- the failure during the merge process indicates that there is a conflict between the local branch and the branch being merged
- in this case git resolves as much as possible but there are things that have to be resolved manually in the conflicte files
 
git log --merge 
	lists commits that are causing the conflict
git diff
	helps in finding the differences between the states of repositories or files
git checkout 
	used to undo the changes made to the file or for changing branches
git reset --mixed
	used to undo the changes to the working directory and staging area
git merge --abort
	helps in exiting the merge process and returning back to the state before the merging began
git reset
	used at the time of merge conflict to reset the conflicted files to their original state

git fetch :
-----------
- git fetch only downloads new data from a remote repository
- does not integrate any of this new data into your working files
- git fetch can be done anytime to update the remote-tracking branches
	git fetch origin
	git fetch -all

git pull :
----------
- git pull updates the current HEAD branch with the latest changes from the remote server
- it downloads new data and integrates it with the current working files
- it tries to merge remote changes with your local ones
	git pull origin master

git merge :
-----------
- creates an extra merge commit every time you need to incorporate changes
- pollutes your feature branch history

git rebase :
------------
- incorporates all the new commits in master branch
- re-writes the project history by creating brand new commits for each commit in the original branch

- to get a list of files that has been changed in a particular commit 
	git diff-tree-r {commit hash}
	-r flag allows the command to list individual files
	commit hash will list all the files that were changed or added in that commit

Resolving merge conflict using conflict editor :
------------------------------------------------
- under your repository name, click pull requests
- in "Pull Requests" list, click the pull request with a merge conflict that you'd like to resolve
- near the bottom of your pull request, click Resolve conflicts
- Decide if you want to keep only your branch's changes, keep only the other branch's changes or make a brand new change, which may incorporate changes from both branches
- delete the conflict marker <<<<<<<< ======== >>>>>>>> and make changes you want in the final merge
- if you have more than one merge conflict in your file, scroll down to the next set of conflict markers and repeat steps four and five to resolve your merge conflict
- once you have resolved all the conflicts in the file, click Mark as resolved
- if you have more than one file with a conflict, select the next file you want to edit on the left side of the page under "conflicting files" and repeat steps 4 to 7 until you've resolved all of your pull request's merge conflict
- once you've resolved all your merge conflicts, click Commit merge. This merges the entire base branch into your head branch
- to merge your pull request, click Merge pull request
